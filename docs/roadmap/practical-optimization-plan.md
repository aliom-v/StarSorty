# StarSorty 实用性优化实施方案（MVP）

## 1. 目标与原则

本阶段不追求“功能多”，只追求“每天都用得上”。

优化目标（按优先级）：

1. 更快找到收藏项目（找得到）
2. 更少信息负担（看得完）
3. 更低遗忘率（记得用）
4. 更高决策效率（留/删更快）

实施原则：

- 先做高频场景，避免大而全
- 优先复用现有 FastAPI + Next.js 架构
- 每个功能必须有可量化验收指标

## 1.1 实施进展（截至 2026-03-05）

- ✅ F1 智能检索基础能力已完成：
  - `GET /search/semantic` 已上线
  - 前端已支持“关键词 / 智能检索”切换
  - 已支持混合排序（语义 + 关键词 + 新鲜度 + 热度）
  - 已支持结果可解释原因（`semantic_reasons`）
  - 已支持语义检索缓存（TTL + 容量可配置）
- ✅ `embedding_backfill` 任务已完成：
  - `POST /embedding/backfill` 可触发回填
  - 支持断点续跑与任务重试（`/tasks/{task_id}/retry`）
- ⏳ F2 / F3 / F4 尚未开始实现

## 2. 当前痛点（结合现有系统）

基于当前能力（同步、分类、标签筛选、搜索），主要缺口：

- 搜索主要依赖关键词和标签，对“模糊记忆”支持不足
- Star 数量增长后，用户难以决定“今天看什么”
- 收藏后长期未回顾，知识资产沉没
- 缺少“是否值得保留”的辅助判断

## 3. MVP 功能范围（第一阶段）

### F1. 智能检索（自然语言 + 语义召回）

用户价值：输入“我上周收藏的 Go 爬虫库”也能找到。

实现范围：

- 新增语义字段：`summary_embedding`
- 新增检索接口：`GET /search/semantic?q=...&limit=...`
- 前端在首页搜索框提供“关键词/智能检索”切换
- 保留当前关键词搜索作为兜底

非目标（本阶段不做）：

- 多向量模型切换
- 混合重排（复杂 rerank）

验收标准：

- Top 10 结果中，人工评估相关项目命中率 >= 70%
- 单次查询 P95 响应时间 <= 1.5s（本地 5k 仓库规模）

### F2. 每周精读清单（Digest）

用户价值：不用自己翻，系统给出“本周最值得看的 10 个”。

实现范围：

- 新增接口：`GET /digest/weekly?limit=10`
- 评分因子（简单版）：
  - 最近 star 时间
  - 仓库活跃度（pushed_at）
  - star/fork 基础热度
  - 分类覆盖度（避免全是同类）
- 前端首页新增“本周精读”卡片区

非目标：

- 邮件推送
- 个性化推荐模型

验收标准：

- 列表生成时间 <= 1s
- 至少 80% 条目具备摘要和分类

### F3. 回顾提醒（30/60/90 天）

用户价值：避免收藏后长期不看。

实现范围：

- 新增“最后访问时间”字段：`last_viewed_at`
- 用户打开仓库详情页时更新该字段
- 新增提醒接口：`GET /review/reminders?days=30,60,90`
- 前端设置页新增提醒阈值开关（本地存储即可）

非目标：

- 系统通知推送
- 多端同步提醒设置

验收标准：

- 详情页打开后 1 秒内写入访问时间
- 提醒列表可按阈值正确过滤

### F4. 保留决策分（Keep Score）

用户价值：快速判断“保留/归档”。

实现范围：

- 新增计算字段：`keep_score`（0-100）
- 评分因子（可解释）：
  - 最近活跃度
  - star 趋势（先用静态近似）
  - README 完整度（长度 + 结构）
  - 是否已分类/有摘要
- 前端详情页展示分数与解释文案

非目标：

- 自动取消 Star（不做危险操作）

验收标准：

- 100% 可见评分解释（至少 3 条因子）
- 同仓库重复计算偏差 <= 5 分

## 4. 数据与接口改动

## 4.1 数据库（建议）

在 `repos` 表新增字段：

- `summary_embedding` TEXT NULL（先 JSON 字符串存储）
- `last_viewed_at` DATETIME NULL
- `keep_score` INTEGER NULL
- `keep_score_reason` TEXT NULL

新增索引：

- `idx_repos_last_viewed_at`
- `idx_repos_keep_score`

## 4.2 API 设计（新增）

- `GET /search/semantic`
  - 入参：`q`, `limit`
  - 出参：`repos[]`, `score`
- `GET /digest/weekly`
  - 入参：`limit`
  - 出参：`items[]`, `reason`
- `GET /review/reminders`
  - 入参：`days`
  - 出参：`groups[{threshold, repos[]}]`
- `POST /repos/{full_name}/viewed`
  - 行为：写入 `last_viewed_at=now()`
- `POST /repos/{full_name}/keep-score/recalculate`
  - 行为：重算单仓库评分

## 4.3 后台任务

新增两个可复用任务：

- `embedding_backfill`：为已有仓库补齐 embedding
- `keep_score_refresh`：周期重算保留分

通过现有 scheduler 触发，保持与当前任务体系一致。

## 5. 前端页面改动

首页（`web/app/page.tsx`）：

- 搜索条增加“智能检索”切换
- 新增“本周精读”模块
- 结果卡片可显示 `keep_score`

详情页（`web/app/repo/*`）：

- 展示保留决策分与解释
- 页面加载后上报 `viewed`

设置页（`web/app/settings/page.tsx`）：

- 新增回顾提醒阈值配置（30/60/90）
- 新增“重建语义索引”按钮（管理员）

## 6. 两周实施排期

### 第 1 周

1. D1-D2：数据库迁移 + API 框架（空实现）
2. D3-D4：语义检索（F1）可用
3. D5：前端检索切换 + 联调

### 第 2 周

1. D1-D2：每周精读（F2）
2. D3：回顾提醒（F3）
3. D4：保留决策分（F4）
4. D5：验收、性能压测、文档补齐

## 7. 验收清单（发布门槛）

- 功能验收：F1~F4 全部可在 UI 完整闭环
- 性能验收：
  - 搜索 P95 <= 1.5s
  - Digest 接口 <= 1s
- 可靠性验收：后台任务失败可重试，失败信息可见
- 可运维验收：关键任务有状态查询接口

## 8. 实施顺序建议

严格按顺序推进，避免并行过多导致复杂度上升：

1. F1 智能检索
2. F2 每周精读
3. F3 回顾提醒
4. F4 保留决策分

理由：F1/F2 直接提升“日常打开频率”，优先收益最高。

## 9. 风险与应对

- 向量检索依赖模型调用成本
  - 应对：先用小模型 + 缓存 embedding
- 分数规则主观性强
  - 应对：先展示解释，支持后续调权重
- 数据迁移影响旧数据
  - 应对：提供 backfill 任务，允许渐进补齐
