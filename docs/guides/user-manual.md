# 用户手册

本文档面向日常使用 StarSorty 的用户，覆盖首页检索、详情页、管理台、设置页以及常见操作流程。

## 适用对象

- 想把 GitHub Star 整理成知识库的个人开发者
- 负责维护 StarSorty 实例的管理员
- 需要查看仓库分类、标签和摘要的协作者

## 使用前准备

在开始前，请确保：

- 服务已经启动
- 可以访问 Web：`http://localhost:1234`
- 已完成至少一次同步
- 若需要执行写操作，已经拿到 `ADMIN_TOKEN`

首次初始化建议参考：`README.md` 和 `docs/guides/deployment-operations.md`

## 页面导航

### 首页 `/`

首页用于检索、筛选和浏览仓库，是最常用的入口。

你可以在首页完成：

- 关键词搜索
- 按语言筛选
- 按分类、子分类筛选
- 按标签筛选
- 按 Star 数过滤
- 切换排序方式
- 查看统计概览
- 打开仓库详情页或 GitHub 原仓库

### 仓库详情页 `/repo`

详情页用于查看单个仓库的完整信息，常见内容包括：

- 基础元信息
- 分类与标签
- 中文摘要或描述
- 搜索命中原因
- Star、更新时间等指标
- README 摘要与人工覆盖后的结果

### 管理台 `/admin`

管理台面向管理员，主要包含：

- 管理员认证
- 手动同步
- 手动分类
- 失败仓库处理
- 导出 Obsidian
- 运行设置修改
- 个性化配置与训练样本查看

### 设置页 `/settings`

设置页用于查看当前实例的公开前端配置状态，例如：

- 分类模式
- 自动分类开关
- GitHub 聚合模式

更完整的运行配置、token 配置状态、失败仓库、用户偏好和兴趣画像均已收口到管理员接口，需要在 `/admin` 中访问。

## 常用操作流程

### 1. 首次使用

推荐顺序：

1. 打开首页确认服务可访问
2. 进入管理台并录入管理员 token
3. 手动触发一次同步
4. 触发后台分类
5. 返回首页检索结果是否出现分类、标签和摘要

### 2. 搜索和筛选仓库

在首页常见的使用方式：

- 输入关键词搜索仓库名称、描述、摘要或命中内容
- 按标签缩小范围，例如 AI、Agent、RAG、CLI 等
- 按语言或最少 Star 数过滤
- 使用相关度排序查找更接近当前需求的仓库

建议：

- 已知目标仓库时，优先用关键词搜索
- 做主题探索时，优先用标签 + 相关度排序
- 收藏仓库很多时，结合语言、标签和 Star 数过滤

### 3. 查看仓库详情

当你在列表页看到感兴趣的仓库时，可以：

- 点击仓库名直接跳转 GitHub
- 点击 `details` 进入详情页
- 在详情页确认分类、标签、摘要和更新时间
- 判断是否需要手动修正分类或补抓 README

### 4. 手动触发同步

适用于：

- 新 Star 了仓库，希望立即同步
- 修改了 GitHub 目标用户配置
- 怀疑数据与 GitHub 不一致

操作方式：

- 进入 `/admin`
- 使用同步区域触发 `Sync`
- 等待任务 ID 返回
- 在首页或接口状态中确认同步完成

也可以直接调用 API：

```bash
curl -X POST "http://localhost:4321/sync" \
  -H "X-Admin-Token: <ADMIN_TOKEN>"
```

### 5. 手动触发分类

适用于：

- 新同步的数据还未分类
- 切换了分类模式或规则
- 想强制重跑一批已存在仓库

常见参数：

- `limit`：本次处理多少条
- `concurrency`：后台分类并发数
- `force`：是否强制重跑已分类仓库
- `include_readme`：分类时是否尝试读取 README

建议：

- 日常手动操作使用后台分类
- 大批量重跑时控制并发，避免超出 AI 配额或触发上游限流

### 6. 查看失败仓库

如果部分仓库多次分类失败，可在管理台查看失败列表。

常见原因：

- README 拉取失败
- AI provider 配置缺失
- 上游接口暂时异常
- 仓库信息本身不完整

你可以：

- 先检查 AI 与 GitHub 配置
- 重新抓取 README
- 修复配置后重试分类
- 在必要时重置失败计数

### 7. 人工修正分类结果

当自动分类结果不理想时，可以对单个仓库做人工覆盖。

适合修正的字段：

- `category`
- `subcategory`
- `tags` / `tag_ids`
- `note`

修正后：

- 首页与详情页会优先展示覆盖结果
- 系统会保留覆盖历史
- 这些人工修改也可能作为训练样本沉淀

### 8. 使用个性化功能

管理台支持按 `user_id` 查看和编辑个性化配置。

你可以维护：

- `tag_mapping`：把系统标签映射成更符合个人习惯的标签
- `rule_priority`：调整规则优先级
- `interest profile`：查看近期兴趣主题画像
- `training samples`：查看从人工覆盖沉淀出来的训练样本
- `few-shot`：查看可用于提示词增强的数据样本

适用场景：

- 同一实例服务多个用户
- 希望不同用户对同类仓库有不同标签偏好
- 想观察搜索行为和点击反馈是否在形成兴趣画像

### 9. 导出到 Obsidian

如果你想把 Star 列表做成知识库笔记，可以使用导出功能。

支持：

- 按标签过滤导出
- 按语言过滤导出
- 下载 ZIP 包后导入 Obsidian

适合场景：

- 做个人技术资料库
- 做专题仓库归档
- 将收藏转成可离线浏览的文档集

## 推荐使用习惯

- 定时同步开启，日常只在需要时手动补同步
- 大量新仓库导入后，再统一触发后台分类
- 搜索时优先使用标签和相关度排序
- 对高价值仓库及时做人工修正和备注
- 定期导出 Obsidian 包或备份 `data/`

## 常见问题

### 为什么首页看不到分类结果？

优先检查：

- 是否已完成同步
- 是否已执行分类
- AI provider 是否配置正确
- 当前分类模式是否为 `rules_only` 且规则未命中

### 为什么写操作失败？

优先检查：

- 是否已配置 `ADMIN_TOKEN`
- 请求头里是否带了正确的 `X-Admin-Token`
- 是否触发了接口限流

### 为什么搜索结果很少？

优先检查：

- 关键词是否过窄
- 标签筛选是否过多
- `tag_mode` 是否使用了 `and`
- 当前实例是否只同步了少量用户或仓库

### 为什么分类任务一直在跑？

优先检查：

- `classify/status` 中的 `remaining` 和 `failed`
- AI provider 是否变慢或被限流
- 并发配置是否过高
